# Технологии Программирования · Архитектурная модель приложения

Приложение реализовано по паттерну **MVVM (Model–View–ViewModel)**, который является одним из базовых подходов к организации слоя представления в Android-разработке [file:3]. В рамках данной лабораторной работы роли компонентов распределяются следующим образом:

- **Model / Data Layer** — классы `Note`, `NoteDao`, `NoteDatabase`, `NoteRepository`, отвечающие за хранение и предоставление данных [file:3].
- **View** — composable-функции `NotesScreen` и `AddEditNoteScreen`, отвечающие за отображение текущего состояния и обработку пользовательских действий [file:3].
- **ViewModel** — класс `NotesViewModel`, инкапсулирующий бизнес-логику, операции доступа к данным и текущее состояние экрана [file:3].

Такое разделение делает проект более поддерживаемым, тестируемым и устойчивым к изменениям. Compose-представление не работает напрямую с базой данных, а получает только уже подготовленное состояние из `ViewModel`, что соответствует современным рекомендациям Android-разработки [file:3].

# Технологии Программирования · Технический разбор `NoteDao.kt`

Класс `NoteDao.kt` представляет собой интерфейс доступа к таблице заметок, аннотированный `@Dao`. Room анализирует данный интерфейс на этапе компиляции и генерирует его реализацию автоматически, что избавляет разработчика от ручной работы с `Cursor`, SQL-соединениями и преобразованием строк таблицы в объекты Kotlin [file:3].

Главная архитектурная особенность данного DAO — метод `getAllNotes()` возвращает не обычный список, а `Flow<List<Note>>`. Это означает, что выборка заметок организована как **реактивный поток**: после любой операции вставки, обновления или удаления Room автоматически отслеживает изменение таблицы `notes` и повторно эмитирует обновлённый список подписчикам [file:3]. Такой механизм особенно важен для Compose, поскольку позволяет интерфейсу автоматически обновляться без ручного вызова повторного запроса.

```kotlin
package com.example.notesapp.data

import androidx.room.*
import kotlinx.coroutines.flow.Flow

@Dao
interface NoteDao {

    @Query("SELECT * FROM notes ORDER BY id DESC")
    fun getAllNotes(): Flow<List<Note>>

    @Insert
    suspend fun insertNote(note: Note)

    @Update
    suspend fun updateNote(note: Note)

    @Delete
    suspend fun deleteNote(note: Note)

    @Query("SELECT * FROM notes WHERE id = :id")
    suspend fun getNoteById(id: Int): Note?
}
```

### Ключевые свойства реализации `NoteDao`

- Метод `getAllNotes()` возвращает **холодный поток** `Flow`, который начинает испускать значения только после появления подписчика [file:3].
- Повторная эмиссия выполняется Room автоматически, поэтому разработчику не требуется вручную инициировать обновление списка после `insert`, `update` или `delete` [file:3].
- Методы `insertNote`, `updateNote`, `deleteNote`, `getNoteById` объявлены как `suspend`, поскольку работа с БД должна происходить в корутине и не должна выполняться на главном потоке [file:3].
- Использование Room обеспечивает compile-time проверку SQL-запросов, что значительно снижает вероятность ошибок времени выполнения [file:3].

# Технологии Программирования · Технический разбор `NotesViewModel.kt`

`NotesViewModel.kt` является центральным звеном взаимодействия между пользовательским интерфейсом и слоем данных. Именно `ViewModel` получает поток заметок из репозитория, преобразует его в пригодное для Compose состояние и предоставляет функции для создания, удаления и обновления записей [file:3].

Ключевой технический интерес представляет применение оператора `stateIn`, который преобразует холодный поток `Flow<List<Note>>` в горячий `StateFlow<List<Note>>`. Это позволяет хранить последнее актуальное состояние списка заметок внутри `ViewModel` и выдавать его новым подписчикам немедленно после подписки [file:3].

```kotlin
package com.example.notesapp.ui

import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import com.example.notesapp.data.Note
import com.example.notesapp.data.NoteRepository
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.flow.SharingStarted
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.stateIn
import kotlinx.coroutines.launch

class NotesViewModel(
    private val repository: NoteRepository
) : ViewModel() {

    val notes: StateFlow<List<Note>> = repository.allNotes
        .stateIn(
            scope = viewModelScope,
            started = SharingStarted.WhileSubscribed(5_000L),
            initialValue = emptyList()
        )

    fun addNote(title: String, content: String) {
        viewModelScope.launch(Dispatchers.IO) {
            val note = Note(
                title = title,
                content = content,
                createdAt = System.currentTimeMillis()
            )
            repository.insertNote(note)
        }
    }

    fun deleteNote(note: Note) {
        viewModelScope.launch(Dispatchers.IO) {
            repository.deleteNote(note)
        }
    }

    fun updateNote(id: Int, title: String, content: String) {
        viewModelScope.launch(Dispatchers.IO) {
            val existing = repository.getNoteById(id) ?: return@launch
            repository.updateNote(existing.copy(title = title, content = content))
        }
    }
}
```

### Разбор ключевых решений в `NotesViewModel`

#### 1. Преобразование `Flow` в `StateFlow` через `stateIn`

Оператор `stateIn(...)` используется для получения горячего состояния из потока данных. В отличие от обычного `Flow`, который начинает работать заново при каждом новом `collect`, `StateFlow` хранит последнее вычисленное значение в памяти и мгновенно отдаёт его подписчику [file:3]. Это особенно полезно в Compose, где экран может быть пересоздан после конфигурационного изменения.

#### 2. `SharingStarted.WhileSubscribed(5000)`

Параметр `SharingStarted.WhileSubscribed(5000)` означает следующее:

- upstream-поток остаётся активным только пока у него есть подписчики;
- после исчезновения последнего подписчика поток не отменяется мгновенно, а ждёт 5000 мс;
- такая задержка предотвращает лишние пересоздания потока при повороте экрана и кратковременной пересборке UI [file:3].

Иными словами, если пользователь повернул устройство, старый экран уничтожается, новый создаётся почти сразу, и поток продолжает жить без повторного старта. Это повышает эффективность и делает поведение экрана более стабильным.

#### 3. Использование `viewModelScope.launch(Dispatchers.IO)`

Все операции записи и чтения, связанные с базой данных, выполняются внутри `viewModelScope.launch(Dispatchers.IO)`. Это означает, что операции с SQLite выносятся в I/O-пул потоков и **не блокируют главный поток**, отвечающий за отрисовку интерфейса [file:3].

Такое решение критически важно для предотвращения зависаний интерфейса и ошибок типа **ANR (Application Not Responding)**. Room также не рекомендует выполнять запросы на главном потоке, а в стандартной конфигурации прямо запрещает это [file:3].

<details>
<summary><strong>Вопрос 1. Что такое smart recomposition в Jetpack Compose и почему этот механизм считается важным для производительности?</strong></summary>

**Ответ:**  
Smart recomposition — это механизм интеллектуальной перекомпозиции интерфейса, при котором Jetpack Compose обновляет не весь экран целиком, а только те участки, которые действительно зависят от изменившегося состояния. Если часть интерфейса не использует изменённые данные, Compose пропускает её при перестроении. Такой подход уменьшает объём работы по отрисовке, повышает отзывчивость UI и делает декларативную модель практически эффективной даже в динамических интерфейсах [file:3].

</details>

<details>
<summary><strong>Вопрос 2. В чём концептуальное отличие StateFlow от LiveData?</strong></summary>

**Ответ:**  
`StateFlow` является частью экосистемы Kotlin Coroutines и представляет собой горячий поток состояния, который всегда содержит текущее значение и немедленно отдаёт его новому подписчику. `LiveData` тесно связана с Android Lifecycle и исторически использовалась как основной observable-holder в Android. Концептуально `StateFlow` лучше интегрирован с корутинами, поддерживает широкий набор операторов Flow и проще масштабируется в реактивной архитектуре, особенно при использовании Compose [file:3].

</details>

<details>
<summary><strong>Вопрос 3. Какую роль играет Dispatchers.IO в предотвращении ANR при работе с базой данных?</strong></summary>

**Ответ:**  
`Dispatchers.IO` предназначен для выполнения операций ввода-вывода, включая запросы к базе данных, чтение файлов и сетевые обращения. Если выполнять SQLite-операции на главном потоке, интерфейс перестанет отвечать, поскольку UI-поток будет занят дисковой операцией. При длительной блокировке система Android может показать сообщение ANR. Перенос работы с Room в `Dispatchers.IO` позволяет сохранить отзывчивость интерфейса и соответствует корректной архитектурной практике [file:3].

</details>

<details>
<summary><strong>Вопрос 4. Почему использование Room предпочтительнее прямой работы с SQLiteOpenHelper?</strong></summary>

**Ответ:**  
Room предоставляет более высокий уровень абстракции над SQLite: разработчик описывает таблицы в виде сущностей (`@Entity`), запросы — через DAO, а базу данных — через `@Database`. При этом Room валидирует SQL-запросы на этапе компиляции, автоматически генерирует необходимый код, поддерживает `Flow`, `suspend`-функции и тесно интегрирован с архитектурными компонентами Android. Это делает код чище, безопаснее и проще в сопровождении, чем ручная работа через `SQLiteOpenHelper` [file:3].

</details>

<details>
<summary><strong>Вопрос 5. Почему ViewModel предотвращает потерю данных при повороте экрана?</strong></summary>

**Ответ:**  
`ViewModel` хранится в `ViewModelStore` и не уничтожается при конфигурационных изменениях, таких как поворот устройства. Когда `Activity` пересоздаётся, новый экран получает доступ к тому же экземпляру `ViewModel`, а значит — к уже загруженному состоянию. Благодаря этому повторное чтение из базы и ручное восстановление состояния не требуются, что положительно влияет на UX и делает интерфейс предсказуемым [file:3].

</details>
