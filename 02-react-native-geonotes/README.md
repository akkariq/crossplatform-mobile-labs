# Технологии Программирования · Технический разбор `database.ts`

Файл `src/utils/database.ts` отвечает за работу с локальной базой данных SQLite. База открывается через `SQLite.openDatabase('geonotes.db')`, после чего при старте приложения вызывается функция `initDatabase()`, которая создаёт таблицу `notes`, если она ещё не существует [file:5].

Структура таблицы включает поля `id`, `title`, `content`, `latitude`, `longitude`, `address`, `photoUri` и `createdAt`, что позволяет хранить как содержимое заметки, так и её географическую и мультимедийную информацию [file:5].

```typescript
import * as SQLite from 'expo-sqlite';
import { GeoNote } from '../types';

const db = SQLite.openDatabase('geonotes.db');

export const initDatabase = (): Promise<void> => {
    return new Promise((resolve, reject) => {
        db.transaction(tx => {
            tx.executeSql(
                `CREATE TABLE IF NOT EXISTS notes (
                    id TEXT PRIMARY KEY NOT NULL,
                    title TEXT NOT NULL,
                    content TEXT NOT NULL,
                    latitude REAL NOT NULL,
                    longitude REAL NOT NULL,
                    address TEXT,
                    photoUri TEXT,
                    createdAt INTEGER NOT NULL
                );`,
                [],
                () => resolve(),
                (_, error) => {
                    reject(error);
                    return false;
                }
            );
        });
    });
};

export const getNotes = (): Promise<GeoNote[]> => {
    return new Promise((resolve, reject) => {
        db.transaction(tx => {
            tx.executeSql(
                'SELECT * FROM notes ORDER BY createdAt DESC',
                [],
                (_, { rows }) => {
                    const notes: GeoNote[] = [];
                    for (let i = 0; i < rows.length; i++) {
                        notes.push(rows.item(i));
                    }
                    resolve(notes);
                },
                (_, error) => {
                    reject(error);
                    return false;
                }
            );
        });
    });
};
```

### Особенности реализации

- Работа с SQLite организована через транзакции `db.transaction(...)`, что гарантирует целостность операций [file:5].
- Все операции обёрнуты в `Promise`, что позволяет вызывать их из `async/await`-логики Redux thunks [file:5].
- Выборка заметок сортируется по `createdAt DESC`, поэтому новые записи отображаются в списке сверху [file:5].
- В рамках самостоятельной части лабораторной работы были реализованы также функции `addNote`, `deleteNote` и `updateNote`, завершающие CRUD-модель [file:5].

# Технологии Программирования · Технический разбор `notesSlice.ts`

Файл `src/store/notesSlice.ts` реализует централизованное управление состоянием списка заметок через Redux Toolkit. Основной особенностью является использование `createAsyncThunk`, который позволяет описывать асинхронные операции загрузки, сохранения и удаления заметок как отдельные action creators с автоматической генерацией стадий `pending`, `fulfilled` и `rejected` [file:5].

```typescript
import { createSlice, createAsyncThunk, PayloadAction } from '@reduxjs/toolkit';
import { GeoNote } from '../types';
import * as database from '../utils/database';

export const loadNotes = createAsyncThunk(
    'notes/loadNotes',
    async () => {
        const notes = await database.getNotes();
        return notes;
    }
);

export const saveNote = createAsyncThunk(
    'notes/saveNote',
    async (note: GeoNote) => {
        await database.addNote(note);
        return note;
    }
);

export const removeNote = createAsyncThunk(
    'notes/removeNote',
    async (id: string) => {
        await database.deleteNote(id);
        return id;
    }
);

interface NotesState {
    items: GeoNote[];
    loading: boolean;
    error: string | null;
}

const initialState: NotesState = {
    items: [],
    loading: false,
    error: null
};

const notesSlice = createSlice({
    name: 'notes',
    initialState,
    reducers: {
        clearError: (state) => {
            state.error = null;
        }
    },
    extraReducers: (builder) => {
        builder
            .addCase(loadNotes.pending, (state) => {
                state.loading = true;
                state.error = null;
            })
            .addCase(loadNotes.fulfilled, (state, action: PayloadAction<GeoNote[]>) => {
                state.loading = false;
                state.items = action.payload;
            })
            .addCase(loadNotes.rejected, (state, action) => {
                state.loading = false;
                state.error = action.error.message || 'Failed to load notes';
            })

            .addCase(saveNote.pending, (state) => {
                state.loading = true;
                state.error = null;
            })
            .addCase(saveNote.fulfilled, (state, action: PayloadAction<GeoNote>) => {
                state.loading = false;
                state.items.push(action.payload);
                state.items.sort((a, b) => b.createdAt - a.createdAt);
            })
            .addCase(saveNote.rejected, (state, action) => {
                state.loading = false;
                state.error = action.error.message || 'Failed to save note';
            })

            .addCase(removeNote.pending, (state) => {
                state.loading = true;
                state.error = null;
            })
            .addCase(removeNote.fulfilled, (state, action: PayloadAction<string>) => {
                state.loading = false;
                state.items = state.items.filter(note => note.id !== action.payload);
            })
            .addCase(removeNote.rejected, (state, action) => {
                state.loading = false;
                state.error = action.error.message || 'Failed to delete note';
            });
    }
});

export const { clearError } = notesSlice.actions;
export default notesSlice.reducer;
```

### Ключевые свойства реализации `notesSlice`

- Состояние приложения централизовано и содержит `items`, `loading`, `error`, что позволяет явно управлять состояниями загрузки, успеха и ошибки [file:5].
- `createAsyncThunk` избавляет от ручного написания множества action types и action creators [file:5].
- После сохранения новой заметки она сразу добавляется в `state.items`, а затем список пересортировывается по дате создания [file:5].
- После удаления заметки из БД выполняется синхронное удаление её из Redux-состояния по `id`, благодаря чему UI обновляется без повторной полной загрузки списка [file:5].

<summary><strong>Вопрос 1. В чем заключается архитектура React Native и как работает мост (Bridge)?</strong></summary>

**Ответ:**  
Архитектура React Native основана на разделении приложения на два основных слоя: JavaScript-слой, где исполняется логика приложения и описывается интерфейс, и нативный слой платформы, где реально создаются визуальные компоненты и вызываются возможности устройства. В классической архитектуре между этими слоями работает асинхронный мост (**Bridge**), через который команды из JavaScript сериализуются и передаются в нативную часть, а события от нативного UI возвращаются обратно. Такой подход обеспечивает кроссплатформенность, но может создавать накладные расходы при большом количестве операций обмена данными [file:5].

</details>

<summary><strong>Вопрос 2. Какие преимущества дает использование Redux Toolkit для управления состоянием?</strong></summary>

**Ответ:**  
Redux Toolkit существенно упрощает классический Redux-подход за счёт встроенной генерации reducers, actions и обработки иммутабельных обновлений состояния. Использование `createSlice` сокращает количество шаблонного кода, а `createAsyncThunk` позволяет единообразно описывать асинхронные запросы и автоматически управлять стадиями `pending`, `fulfilled` и `rejected`. В результате код становится компактнее, понятнее и лучше масштабируется при усложнении приложения [file:5].

</details>

<details>
<summary><strong>Вопрос 3. Чем отличается работа с нативными модулями в React Native от нативной разработки?</strong></summary>

**Ответ:**  
В нативной разработке программист напрямую обращается к API платформы через Kotlin/Java или Swift/Objective-C. В React Native доступ к камере, геолокации, файловой системе и другим возможностям осуществляется через готовые библиотеки или мосты, которые скрывают платформенную реализацию за единым JavaScript-интерфейсом. Это упрощает разработку, но в некоторых случаях создаёт дополнительный слой абстракции и ограничивает гибкость по сравнению с полностью нативным кодом [file:5].

</details>

<details>
<summary><strong>Вопрос 4. Как обеспечивается кроссплатформенность кода в React Native?</strong></summary>

**Ответ:**  
Кроссплатформенность достигается за счёт того, что разработчик пишет единый JavaScript/TypeScript-код, который интерпретируется React Native runtime и преобразуется в нативные компоненты Android и iOS. Большая часть бизнес-логики, навигации, работы с состоянием и пользовательских сценариев остаётся общей для обеих платформ, а различия платформ при необходимости скрываются внутри библиотек или платформенных адаптеров. Это позволяет существенно сократить объём дублируемого кода по сравнению с отдельной нативной реализацией [file:5].

</details>

<details>
<summary><strong>Вопрос 5. Почему SQLite подходит для локального хранения заметок в мобильном приложении?</strong></summary>

**Ответ:**  
SQLite является встроенной реляционной базой данных, подходящей для автономного хранения структурированных данных непосредственно на устройстве пользователя. Для приложения заметок она удобна тем, что не требует внешнего сервера, поддерживает SQL-запросы, обеспечивает устойчивое хранение данных между запусками приложения и хорошо подходит для CRUD-сценариев. В сочетании с `expo-sqlite` она позволяет достаточно просто организовать локальный persistence layer в кроссплатформенном приложении [file:5].

</details>
